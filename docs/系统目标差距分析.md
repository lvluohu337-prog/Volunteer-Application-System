# 系统目标差距分析

## 1. 先说结论

当前系统已经具备“真实学生 + 真实招生数据 + 结构化推荐结果 + 正式报告页面展示”的基础，但距离你要的“可直接售卖的正式高考志愿方案系统”还差最后一段最关键的交付链路。

最核心的判断不是“系统什么都没有”，而是：

- 已经能算出一部分正式结果；
- 已经能在页面里展示一部分正式结果；
- 但还不能稳定地把这些结果做成可交付、可下载、可回归测试的正式报告产品。

一句话概括：

当前更像“已经接近成型的正式方案预览系统”，还不是“完整可售卖的正式交付系统”。

## 3. 当前已经实现了什么

### 3.1 主流程入口已经比之前清晰很多

当前系统已经不再把用户一上来分散到大量页面里。

已经完成的部分包括：

- 侧边导航只保留主流程入口。
- 工作台只保留“新增学生 / 继续处理学生 / 生成正式报告”三类核心动作。
- 学生详情页已经被收敛成核心业务入口。
- 报告页面已经能呈现正式推荐结构，而不只是段落摘要。

结论：

- “不好用、找不到入口”的问题已经被改善了一大步。
- 但还没有完全等价于一个完整的向导式单流程产品。

### 3.2 真实招生数据不是空的

从当前项目结构和既有台账看，河南真实数据已经不是样例摆设，而是已经进入正式表结构。

当前已具备的真实数据层包括：

- 院校基础数据
- 专业基础数据
- 一分一段
- 招生计划
- 院校录取分
- 专业录取分
- 选科要求
- 部分院校规则
- 部分风险规则
- 部分政策趋势

结论：

- “有没有真实数据”这个问题，至少对河南来说，答案是有。
- 但“多省份都达到正式可售状态”这个问题，答案还不是。

### 3.3 真实推荐结果已经结构化

当前后端已经不是只输出泛泛段落，而是已经能给出结构化字段：

- `recommendationTable`
- `firstChoice`
- `alternatives`
- `notRecommended`

并且这些字段已经被正式接入报告 JSON。

结论：

- “真实推荐结构不存在”这个说法已经过时。
- 现在的主要问题不是没有结构，而是结构还没有被完整导出成交付物。

### 3.4 报告页已经接上结构化推荐

当前 `ReportsPage.vue` 已经展示了：

- 院校 + 专业 + 专业组/代码推荐区
- 冲 / 稳 / 保分区
- 第一志愿建议
- 备选志愿
- 不建议报考项

结论：

- 这一层已经不是最大缺口。
- 现在最大的缺口已经转移到了导出层和稳定性层。

### 3.5 PDF / DOCX 技术导出存在，但仍是旧形态

当前导出器仍然只围绕这些字段生成段落：

- `reportTitle`
- `reportSubtitle`
- `activeProductLabel`
- `sections`
- `advisorNotes`
- `disclaimer`

也就是说，当前导出是“技术上能生成 PDF / DOCX”，但不是“正式志愿推荐表交付版导出”。

结论：

- 现在不能把“能导出文件”误判成“导出能力已经完成”。

## 4. 当前未实现或未收口的关键能力

### 4.1 还没有正式的结构化报告导出

当前最大短板就是：

- 页面里已经有结构化推荐结果；
- 但 PDF / DOCX 里还没有这套正式推荐表。

具体缺口包括：

- 没有正式导出冲 / 稳 / 保推荐表
- 没有正式导出专业组/代码
- 没有正式导出最低分、最低位次、位次差
- 没有正式导出风险等级和推荐理由
- 没有正式导出第一志愿、备选志愿、不建议项

### 4.2 导出测试还不稳定

当前 `backend/tests/test_report_export.py` 仍然直接导入了 `planning_repository`，并测试 `planning_repository.export_report_package(...)`。

这意味着：

- 导出测试没有彻底隔离成“只测 exporter”
- 测试链路仍然带着报告仓储层
- 一旦仓储层、真实数据层、导出层耦合过深，就容易出现长时间卡住

这也是最近导出测试异常耗时的核心工程风险。

### 4.3 跨省正式能力还不够

当前台账里最明显的数据缺口仍然是：

- `province_batches` 为空
- 目标省份正式支持范围没有写清楚
- 浙江等目标省份没有明确达到与河南同等级别的真实可售状态

这说明现在的正式能力更接近：

- 河南较强
- 其他省份未必正式

### 4.4 专业与城市解释深度还不足以撑高价档

当前虽然已有：

- `major_categories`
- `city_industries`

但它们更像基础方向库，还不是深度交付知识库。

缺少的深层内容包括：

- 具体专业课程难度说明
- 适合什么学生 / 不适合什么学生
- 专业对应的考研路径
- 专业对应的就业路径
- 城市产业和具体专业之间的映射
- 城市迁移路径和就业落地建议

这会直接影响 399 以上版本的价值感，尤其是 699 / 999。

### 4.5 699 / 999 的“深度版能力”还没有产品化

当前产品定义里已经写了 699 / 999 的方向，但实际系统还没有形成完整交付能力。

缺少的能力包括：

- 多方案对比
- 名校优先 / 专业优先 / 城市优先三套策略
- 考研路径分析
- 就业迁移路径分析
- 家庭预算和距离因素
- 顾问人工复核入口

## 5. 分层差距分析

## 5.1 产品流程层

### 已实现

- 主入口比之前清晰
- 用户可以从工作台进入主流程
- 学生详情页已经承担流程工作台角色
- 报告页面已接上正式推荐展示

### 仍有缺口

- 仍不是完整的一体化向导式流程
- `analysis / majors / plan / reports` 之间仍更像串联页面，而不是真正的单流程引导
- demo / base-data / settings 等页面虽然入口弱化了，但仍在代码中存在

结论：

- 产品流程层已经明显改善，但不是当前最大短板。

## 5.2 数据层

### 已实现

- 河南核心招生表结构已经存在
- 真实录取分、专业、计划、位次、选科要求已具备基础能力

### 主要缺口

- `province_batches` 为空
- 风险规则覆盖薄
- 政策趋势覆盖薄
- 多省份正式支持能力不明确
- 深层专业 / 城市解释知识库尚未建立

结论：

- 数据不是“没有”，而是“局部真实、局部不足、跨省不均衡”。

## 5.3 推荐算法层

### 已实现

- 已经不是纯画像文案系统
- 已有真实候选匹配
- 已有结构化推荐结果输出

### 主要缺口

- 综合推荐分和解释维度还没有完整产品化落地
- 城市适配度 / 专业适配度还没有作为强交付字段系统呈现
- 更大范围样本下的回归稳定性还需要验证

结论：

- 当前算法层不是本轮最急的问题。
- 现在更应该先把已有结果稳定导出，而不是立刻重写推荐引擎。

## 5.4 报告导出层

### 已实现

- 技术上可生成 PDF / DOCX
- 可记录导出留痕

### 主要缺口

- 仍是段落型导出
- 没有表格型正式推荐交付
- 没有把结构化推荐结果接入导出
- 没有形成真正的 399 元标准报告成品

结论：

- 这是当前最大的业务缺口。

## 5.5 测试稳定性层

### 已实现

- 已有部分单元测试
- 已有导出相关测试文件

### 主要缺口

- 导出测试未彻底隔离
- 测试容易把仓储层和导出层绑在一起
- 最近已经出现过长时间卡住的问题
- 产物目录缺少更明确的测试生成物管理策略

结论：

- 这是当前最大的工程缺口。

## 5.6 合规边界层

### 已实现

- 画像层和真实录取判断层在设计上已有边界
- 报告中已有免责声明意识

### 主要缺口

- 还需要统一页面、报告 JSON、导出文件中的合规表达
- 还需要系统检查是否存在容易被误解为承诺录取的描述

结论：

- 合规不是当前最堵主链路的点，但必须在导出正式化之前收口。

## 6. 距离目标效果还差什么

### 距离 399 元正式报告，还差 5 件事

1. 结构化推荐表正式进入 PDF / DOCX
2. 导出测试彻底隔离，避免长时间卡住
3. `province_batches` 和正式支持省份清单补齐
4. 风险规则和政策解释补强
5. 专业 / 城市解释层提升到“可售卖”的深度

### 距离 699 / 999 深度报告，还差 5 件事

1. 多方案对比能力
2. 多策略版本能力
3. 考研路径分析
4. 就业 / 城市迁移路径分析
5. 顾问人工复核与深度品牌化交付

---

## 7. 当前最应该先做什么

如果只按“离正式售卖最近”的角度排序，当前最应该先做的是：

1. 导出链路隔离与收口
2. 结构化推荐表导出
3. `province_batches` 和正式省份支持范围核验
4. 本地知识解释库建设
5. 深度报告能力建设

其中第一优先级不是再改推荐算法，而是：

`把已经算出来、已经在页面上显示出来的正式推荐结果，稳定地导出成真正可交付的正式报告。`

## 8. 建议新增的下一轮任务方向

下一轮建议单独开一个“导出链路隔离与正式化”任务，范围控制为：

- 不碰推荐算法
- 不碰大前端重构
- 只处理 `backend/report_exporters.py`
- 只处理 `backend/tests/test_report_export.py`
- 只处理最小 mock 报告数据
- 只解决结构化推荐表导出和测试稳定性

之后再开下一轮任务去做：

- `province_batches`
- 目标省份正式支持范围
- 专业 / 城市 / 路径解释知识库

---

## 9. 这次文档更新后的结论

这次复核后，当前系统的真实状态可以概括为：

- 主流程入口：比之前好很多，已经不是最主要问题。
- 后端推荐结构：已经成型，不是当前最急的问题。
- 报告页面展示：已经成型，不是当前最急的问题。
- 导出层：仍然是当前最大的业务短板。
- 导出测试稳定性：仍然是当前最大的工程短板。

所以本轮最重要的不是“继续做更多页面”，而是把交付链路收口。
