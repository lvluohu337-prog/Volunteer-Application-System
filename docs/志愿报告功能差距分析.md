# 志愿报告功能差距分析

## 1. 结论先看

当前项目已经不是“纯演示壳子”，而是具备了：

- 学生档案录入；
- 画像辅助分析；
- 河南真实招生数据入库；
- 基于分数/位次/选科的真实候选匹配；
- 前端报告预览；
- PDF / Word 技术导出。

但距离“399 元可售卖高考志愿方案报告”仍有明显差距。当前系统的核心问题不是完全没有数据，而是：

1. 已有真实招生数据，但最终报告没有输出成正式的“院校 + 专业 + 专业组/代码”推荐表。
2. 已有候选匹配能力，但冲稳保分层和推荐排序还不够产品化，无法稳定产出 3/5/3 的正式方案。
3. PDF/Word 导出仍是段落型文本导出，不是可交付的正式表格化报告。

一句话判断：当前系统更像“有真实数据底座的分析预览版”，还不是“可直接卖 399 元的正式志愿方案交付版”。

## 3. 当前已实现功能清单

### 3.1 学生档案层

已实现：

- 学生基础信息录入：姓名、性别、省份、学校、届别、选科、批次、目标方向等。

- 第 2 条目标“性格/前六字/星座只能作为解释层”在设计上已经有明确边界。

已实现：

- `backend/intake_inference.py` 能根据生日/时辰推导星座、四柱、五行倾向、兴趣方向、区域倾向、发展目标。
- `backend/planning_repository.py` 中的 `_build_portrait_major_recommendation()` 已把画像推荐限制为辅助解释，不直接参与录取概率判断。
- 专业页已有：
  - 推荐方向；
  - 谨慎方向；
  - 画像解释；
  - 家长诉求匹配。

判断：

- 画像辅助能力已经成型。
- 这部分更接近“解释层”，而不是“正式填报层”，方向正确。

### 3.3 基础专业 / 城市知识库

## 5. 数据层缺口

## 5.1 已有数据

- 河南真实院校专业录取数据：有。
- 河南一分一段：有。
- 河南招生计划：有。
- 院校基础信息：有。
- 专业基础信息：有。
- 部分选科要求：有。
- 部分风险规则：有。
- 部分政策趋势：有。

## 5.2 明显缺口

### A1. 目标省份覆盖不足

- 河南真实数据已接通。
- 浙江学生会退回到通用兜底方案，说明浙江真实招生表尚未接通到同等能力。
- 用户点名的“浙江 / 河南等目标省份”，目前至少浙江不是正式可售状态。

### A2. 专业组代码数据虽已入库，但未成为交付主字段

- 真实查询里已经取到了 `plan_group_code`。
- 但正式页面、正式报告、PDF/Word 导出都没有展示。

### A3. 代码类字段未进入交付层

- 已能查到：
  - `institution_code`
  - `major_code`
  - `plan_group_code`
- 但没有变成最终报告表格列。

### A4. 城市产业与专业就业路径映射不够深

- `city_industries` 当前更像“城市画像库”。
- 还没有形成“城市产业 -> 具体专业 -> 实习/就业/考研路径”的细粒度映射。
- 还不足以支撑高价报告的强解释力。

### A5. 专业库粒度偏粗

- 当前 `major_categories` 偏“专业大类”。
- 缺少系统化的“专业大类 -> 具体专业 -> 课程难度 -> 就业方向 -> 考研方向”的结构。

### A6. 风险规则库偏稀疏

- `admission_risk_rules` 只有 14 条。
- `policy_trends` 只有 9 条。
- `institution_rules` 只有 133 条。
- 对正式商业交付来说，覆盖面还不够。

## 6. 推荐算法缺口

## 6.1 当前算法的优点

- 已从“纯文案推荐”升级到“真实招生表候选匹配”。
- 已有选科过滤。
- 已有位次/分数门槛。
- 已有计划风险和录取概率摘要。

## 6.2 当前算法的关键短板

### B1. 冲稳保分层不稳定

实测真实学生时：

- 命中候选 180 条；
- 但全部落在 `rush`；
- `steady` 和 `safe` 为空。

这意味着当前冲稳保分层逻辑虽然存在，但还不具备正式交付可用性。

### B2. 真实推荐排序没有接入专业适配度 / 城市适配度

真实候选分支中：

- `keyword_hits` 为空；
- `preference_score` 固定为 0；
- 综合分主要由选科、位次、分数、概率、计划稳定度构成。

结果：

- 当前更像“硬条件筛选器”；
- 还不是“面向交付的综合推荐引擎”。

### B3. 没有正式的综合推荐分模型

缺少面向交付的统一打分：

- 位次差分；
- 分数差分；
- 专业匹配度；
- 城市匹配度；
- 计划稳定度；
- 院校层次偏好；
- 调剂接受度；
- 家长诉求匹配度。

### B4. 没有稳定的数量配额与兜底策略

目标要求至少：

- 冲刺 3 个；
- 稳妥 5 个；
- 保底 3 个。

当前实际情况：

- 真实分支每栏最多只取 3 个；
- 某些栏可能为 0；
- 没有 fill-down / fill-up 机制；
- 没有“数量不够时如何回补”的策略。

### B5. 没有真实学生版第一志愿策略器

- 第一志愿建议只在 demo 报告链路中出现。
- 真实学生正式报告链路没有独立的 first choice 生成逻辑。

### B6. 没有真实学生版“不建议项”

- 当前只有画像层面的 `avoidMajorDirections`。
- 没有真实招生维度的：
  - 不建议冲的院校；
  - 不建议报的专业组；
  - 不建议接受的高风险方向。

### B7. 缺少排序去重优化

- 专业页会把真实候选按专业名称聚合成卡片。
- 报告并没有产出最终志愿表去重逻辑。
- 还缺少“同校多组、多专业、多批次”的正式整理规则。

### B8. 缺少一致性校验

当前案例中出现：

- 位次看起来较优；
- 但参考最低分明显更高；
- 结果大量候选全落在冲刺。

说明仍缺少：

- 分数与位次一致性校准；
- 年份口径统一；
- 省份一分一段对齐；
- 数据异常时的保守降权。

## 8. PDF 报告展示缺口

### D1. 当前 PDF / Word 是段落导出，不是表格导出

`backend/report_exporters.py` 当前做法是：

- 把 `reportTitle`、`sections`、`advisorNotes`、`disclaimer` 组装成文本块；
- 再导出为 PDF / DOCX。

没有表格模型，也没有列定义。

### D2. 报告页预览也是段落型

报告页主要是：

- 模块卡片；
- 摘要卡片；
- `section.body` 段落渲染。

没有正式推荐表预览，因此导出层也拿不到结构化表格数据。

### D3. 冲稳保没有版式化分区表

- 有栏目概念；
- 没有正式表格。

### D4. 99 / 399 / 999 内容差异还停留在模板标签层

- 当前有产品配置和模板映射。
- 但交付内容差异还不够清晰。
- 更像“模块标签不同”，而不是“正式报告能力不同”。

### D5. 报告品牌感和正式交付结构不足

- 还没有真正的封面、摘要页、核心推荐表、风险页、家长沟通页、结论页等交付结构。

## 9. 距离 399 元报告还差什么

### 必须补齐的能力

1. 真实学生正式输出“院校 + 专业 + 专业组/代码”推荐表。
2. 每条推荐项显示：
   - 院校；
   - 专业；
   - 专业组代码；
   - 最低分；
   - 最低位次；
   - 位次差；
   - 风险等级；
   - 计划人数；
   - 城市；
   - 推荐理由。
3. 冲稳保必须稳定成形，至少达到 3 / 5 / 3。
4. 生成第一志愿建议与备选志愿建议。
5. 生成不建议方向 / 不建议项。
6. 正式 PDF / Word 导出必须能展示表格。
7. 浙江等目标省份需要补齐真实数据接入，否则无法卖跨省正式报告。

### 399 元价值感提升项

- 城市产业解释要落到具体专业与未来路径。
- 风险提示要具体到“调剂 / 计划波动 / 专业热度 / 选科限制”。
- 家长沟通建议要从段落升级为“决策优先级建议”。

#### 第二步：升级真实匹配排序逻辑

- 增加表格渲染；
- 输出正式“冲 / 稳 / 保推荐表”。

#### 第五步：扩目标省份数据

- 优先补浙江；
- 然后再补其他目标省份。

4. 最后补“浙江等目标省份数据接入”。

原因：

- 先有稳定推荐结果，展示和导出才有意义。
- 先做导出但不改推荐结构，只会导出一份“更漂亮的泛文本”。

## 14. 当前系统最核心的短板

最核心短板不是“没有数据”，而是：

### 1. 真正可卖的核心推荐表还没打通

数据已经能查到，报告却还没把它做成正式推荐表。

### 2. 冲稳保算法还不够产品化

当前真实候选可能全部挤在“冲”，无法稳定产出正式方案。

### 3. 导出层还是文本导出，不是正式方案导出

PDF / Word 现在更像“把网页摘要打印出来”，还不是“交付版志愿方案”。
