# 数据整理与正式版落地清单

基于当前项目代码、数据库结构和目标文档，数据整理工作的验收标准不是“文件分类完成”，而是：

- 数据能稳定导入数据库
- 推荐、匹配、风险、报告链路能跑通
- 正式版关键省份能产出可信结果
- 后续年度数据能按同一流程持续更新

主要依据：

- [高考志愿智能决策辅助系统_项目总目标说明](../高考志愿智能决策辅助系统_项目总目标说明.md)
- [系统目标差距分析](./系统目标差距分析.md)
- [backend/database.py](../backend/database.py)
- [backend/admissions_engine.py](../backend/admissions_engine.py)
- [backend/rules_engine.py](../backend/rules_engine.py)

## 一、总目标

最终需要达到两层效果。

### 1. 系统可用

- [ ] 能录入学生信息
- [ ] 能基于真实数据生成志愿方案
- [ ] 能展示冲稳保、院校专业建议、基本风险提示
- [ ] 能生成报告

### 2. 正式版系统可用

- [ ] 核心省份数据完整
- [ ] 关键规则口径准确
- [ ] 导入流程可重复执行
- [ ] 推荐结果可解释、可复核、可更新
- [ ] 报告适合对外正式交付

## 二、数据整理总清单

### 1. 盘点现有数据

- [ ] 统计 `data-needed-standardized` 每个分类下的文件数、格式、年份、省份
- [ ] 标记每个文件属于哪类数据：院校、专业、计划、投档线、一分一段、批次线、选科、政策、章程
- [ ] 建立主清单：`文件路径 -> 数据类型 -> 省份 -> 年份 -> 来源 -> 可导入性`

交付物：

- `data_inventory.csv`

### 2. 确定权威源优先级

- [ ] 同一类数据建立优先级：官方 Excel > 官方 PDF > 整理版 Excel > 图片截图
- [ ] 同一省份、同一年、同一口径只保留一个正式主源
- [ ] 其余文件标记为补充源、校验源、弃用源

交付物：

- `data_source_priority.csv`

### 3. 定义目标数据模型

- [ ] 列出项目真正需要的表
- [ ] 为每张表定义最小字段集、主键、唯一约束、外键关系
- [ ] 明确哪些字段是正式版必填，哪些是增强项

建议优先锁定这些表：

- `institutions`
- `majors`
- `province_batches`
- `score_segments`
- `admission_plans`
- `institution_admission_scores`
- `major_admission_scores`
- `subject_requirements`
- `institution_rules`
- `admission_risk_rules`
- `policy_trends`

交付物：

- `data_dictionary.xlsx` 或 `schema_mapping.md`

### 4. 建立文件到表的映射

- [ ] 每个文件明确映射到哪张表
- [ ] 标记能否直接导入
- [ ] 标记缺哪些字段
- [ ] 标记是否需要 OCR、PDF 抽表或人工补录

交付物：

- `file_to_table_mapping.csv`

### 5. 统一口径和枚举

- [ ] 统一省份名称
- [ ] 统一年份格式
- [ ] 统一科类、选科口径，如“物理类 / 首选物理 / 理工”
- [ ] 统一批次口径，如“本科批 / 本科普通批 / 本科一批”
- [ ] 统一院校名称、院校代码、专业名称、专业代码

交付物：

- `normalization_rules.yaml`

### 6. 拆分可直接导入和待清洗数据

- [ ] Excel、CSV 放入 `ready_to_import`
- [ ] PDF 放入 `needs_extract_pdf`
- [ ] 图片放入 `needs_ocr`
- [ ] 讲义、宣传、解释性材料放入 `reference_only`

交付物：

- 一套规范目录结构
- 一份状态标签清单

### 7. 做结构化抽取

- [ ] 先处理 `05_admission_plans`
- [ ] 再处理 `03_score_segments`
- [ ] 再处理 `04_admissions_scores`
- [ ] 最后处理 `02_batches_policies` 和 `06_policy_rules`

处理原则：

- 先吃表格
- 再吃 PDF
- 最后才碰图片

### 8. 数据清洗

- [ ] 去重
- [ ] 补齐缺失字段
- [ ] 纠正明显错码、错列、错年份
- [ ] 处理同校不同名、同专业不同写法
- [ ] 处理院校专业组和普通专业口径差异

### 9. 入库脚本建设

- [ ] 每类数据单独做导入脚本
- [ ] 支持重复执行
- [ ] 支持校验失败回滚
- [ ] 记录导入日志、错误记录、跳过记录

### 10. 数据质量校验

- [ ] 行数校验

- [ ] 代码匹配校验
- [ ] 年份、省份、批次一致性校验

### 12. 正式版验收

- [ ] 选定 1 到 3 个重点省份先跑通正式链路
- [ ] 用真实学生样例做回放测试
- [ ] 抽查推荐结果是否符合人工判断

- [ ] 确认后再扩省份

## 三、建议执行顺序

### 第一阶段：把底座做实

- [ ] 院校主数据
- [ ] 专业主数据
- [ ] 招生计划
- [ ] 一分一段
- [ ] 院校、专业录取线
- [ ] 批次线

结果：

- 系统开始具备基础推荐能力

### 第二阶段：把风险做实

- [ ] 选科要求
- [ ] 章程规则
- [ ] 退档、调剂、体检、语种规则
- [ ] 政策趋势

结果：

- 推荐从“能算”变成“更可信”

### 第三阶段：把正式交付做实

- [ ] 报告结构统一
- [ ] 数据解释口径统一
- [ ] 导入流程自动化
- [ ] 更新流程制度化

结果：

- 正式版系统可持续运营

## 五、验收标准

### 达到“系统可用”

- [ ] 至少 1 个省份全链路跑通
- [ ] 学生输入后能生成方案
- [ ] 方案能显示冲稳保
- [ ] 能出报告
- [ ] 核心数据不是演示假数据

### 达到“正式版系统可用”

- [ ] 至少重点目标省份数据完整
- [ ] 每类关键数据都有权威主源
- [ ] 导入脚本可重复执行
- [ ] 数据错误可追踪
- [ ] 推荐结果可人工复核
- [ ] 报告可直接面向真实用户交付
- [ ] 新年度数据能按同流程更新
